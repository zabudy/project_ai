## Обзор проекта

Команда разработки продуктов на основе агентов с многокомпонентной архитектурой. 
Специализированные роли обрабатывают требования, архитектуру, реализацию, 
тестирование, ревью и документацию.

**Технологический стек**: Смотри `PROJECT.md` для языка, рантайма, фреймворка, базы данных и инструментов.

## Настройка

Смотри `PROJECT.md` для инструкций по установке, настройке окружения и командам базы данных.

## Сборка и тестирование

Смотри `PROJECT.md` для полной таблицы команд (dev, build, test, lint, typecheck и т.д.).

**Ключевые компоненты:**

## Стиль кода

- Функции должны иметь одну ответственность и оставаться короткими
- Используй понятные, описательные имена; никаких сокращений
- Следуй порядку импортов, определенному в `PROJECT.md`
- Следуй соглашениям по именованию файлов из `PROJECT.md`
- Смотри `PROJECT.md` для правил кодирования, специфичных для языка

## Тестирование

- Используй фабрики и фикстуры, а не жестко закодированные тестовые данные
- Мокай внешние сервисы (особенно AI API), никогда не вызывай реальные API в тестах
- Каждый критерий приемки из PRD должен иметь хотя бы один тест
- Следуй соглашениям по именованию и расположению тестовых файлов из `PROJECT.md`
- **Особое внимание**: Тестируй обработку ошибок AI сервисов (таймауты, недоступность, невалидные ответы)

## Безопасность

- НИКОГДА не логируй токены, пароли или PII (персональные данные)
- НИКОГДА не коммить `.env`, учетные данные или API ключи (особенно OpenAI ключи!)
- НИКОГДА не отключай middleware безопасности для удобства
- Предотвращение инъекций в соответствии с технологическим стеком (см. `PROJECT.md`)
- Смотри `PROJECT.md` для специфических правил безопасности проекта
- **AI специфика**: Проверяй и санитизируй AI-generated контент перед отображением пользователю

## Git Workflow

- Ветки: `feat/описание`, `fix/описание`, `chore/описание` (например, `feat/ai-summarization`)
- Коммиты: conventional commits (`feat:`, `fix:`, `chore:`, `docs:`, `test:`)
- PR: требуют прохождения CI + человеческого одобрения
- Squash merge в main

## Agent Workflow

Фичи проходят через определенный конвейер:

1. **PRD** создается agent'ом product-manager в `docs/prds/` (используя `create_prd.md`)
2. **Дизайн-документ** создается architect agent'ом в `docs/architecture/`
3. **Реализация** выполняется implementer agent'ом в feature ветке
4. **Тесты** валидируются tester agent'ом (особое внимание тестам AI интеграций)
5. **Ревью** проводится code-reviewer agent'ом
6. **Merge** утверждается человеком

Шаблоны PRD находятся в `docs/templates/create_prd.md`.

## Специфические AI инструкции

### Работа с AI моделями
- Всегда предполагай, что AI сервисы могут быть недоступны или медленны
- Реализуй graceful degradation (достойное ухудшение) при сбоях AI
- Кэшируй запросы к AI где возможно
- Следи за потреблением токенов и стоимостью

### Промпт инжиниринг
- Храни промпты в отдельном модуле, не зашивай в код
- Версионируй промпты
- Документируй ожидаемый формат ввода/вывода для каждого промпта

### Обработка ошибок
- Обрабатывай специфические AI ошибки (rate limits, context length exceeded, moderation flags)
- Предоставляй понятные сообщения пользователю при сбоях AI
- Логируй ошибки AI без раскрытия чувствительных данных

## Коммуникация

- Все обсуждения архитектуры и решений документируй в `docs/architecture/`
- Изменения в требованиях отражай в обновлении PRD
- Технический долг фиксируй в отдельном разделе PROJECT.md или создавай задачу
- При обнаружении проблем с AI интеграцией (качество ответов, стоимость) создавай issue с примерами
